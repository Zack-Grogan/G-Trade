# Development-stage defaults

**When this applies:** Starting work with or without an issue; exploratory vs issue-driven.

**Do this:**
- **Issue exists:** Work issue-aware; use issue-to-pr skill; reference issue in branch, commits, PRs.
- **No issue yet:** Treat as exploration. Do not invent an issue; document discoveries and propose formalizing (Linear or plan) when ready. Still update docs if behavior/config changes; produce PR-ready summary if code changed.
- Allow exploration without blocking on issue creation; encourage converting to tracked work when formalizing.

**Full procedure:** docs/engineering-system/cursor-operating-model.md, docs/engineering-system/linear-workflow.md; skill: issue-to-pr when issue exists.
