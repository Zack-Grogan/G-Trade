## Design Context

### Users
Primary users are hybrid trading operators who need one interface for live execution control, runtime monitoring, and replay/forensics. They are time-sensitive and often running overnight sessions where trust, visibility, and fast intervention matter more than decorative UI.

### Brand Personality
Utilitarian, direct, and operationally confident. The interface should feel like mission control: calm under load, explicit about risk/system state, and optimized for keyboard-first workflows.

### Aesthetic Direction
Low-noise terminal-native UI with strict hierarchy and progressive disclosure. Default view surfaces only critical runtime signals (health, stream, account/risk, decisions, alerts). Secondary diagnostics live behind tabs and searchable panes. Avoid ornamental styling; emphasize clarity, density control, and scanability.

### Design Principles
1. Show operator-critical state first; everything else is one action away.
2. Keep one primary action path per screen and consistent keybindings globally.
3. Prefer structured summaries over verbose logs, with drill-down on demand.
4. Use bounded refresh lanes and background workers to preserve responsiveness.
5. Preserve safe operational guardrails for control actions (confirmations, explicit intent).

### Distilled IA Constraints
- `Overview` is the default landing surface and must be sufficient for 80% of runtime monitoring.
- `Execution`, `Decisions`, `Events`, `Runs`, and `Controls` are explicit tabs; no hidden navigation model.
- Global search/filter shortcuts are always available where lists are shown.
- Status color semantics must remain minimal and consistent (healthy/attention/error).
