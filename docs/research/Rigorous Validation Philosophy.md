# Rigorous Validation Philosophy

*Reference: strategy/research only. Current operator interface is CLI-only; see [../README.md](../README.md) and [../OPERATOR.md](../OPERATOR.md).*

Intraday ES strategies are highly prone to overfitting due to abundant parameters (time windows, filters, pattern triggers) and the noisy, non-stationary nature of high-frequency data.  We therefore treat backtests with **extreme skepticism**.  Our philosophy is to assume any strong in-sample performance might be a statistical fluke unless it survives *aggressive* validation.  This means:

- **Focus on realistic deployment conditions.**  Simulate the live environment as closely as possible: use actual market hours (e.g. CME Globex ES open 6:30–16:15 ET), include realistic transaction costs and slippage, and avoid any use of future data.  Backtests must incorporate granular trade assumptions (fill delays, spread, order-book dynamics) so that results aren’t merely theoretical.【41†L299-L307】【27†L105-L108】.  
- **Emphasize robust metrics.**  Rather than chasing raw profits, we insist on significance tests and multiple metrics.  For example, use the Deflated Sharpe Ratio (DSR) and Probability of Backtest Overfitting (PBO) to judge if a high Sharpe is just luck【39†L66-L73】【40†L64-L72】.  A strategy that looks great in-sample but collapses out-of-sample is probably curve-fitted【36†L179-L187】.  In practice we expect *some* performance degradation when moving out-of-sample (a smoothly lower Sharpe is normal), but a sudden collapse or one-hit wonder (e.g. Sharpe >1.5 only in-sample) is a red flag【33†L98-L107】【36†L179-L187】.  
- **Multi-modal, regime-aware mindset.**  Intraday market regimes (quiet vs. volatile, trending vs. mean-reverting) change rapidly.  We favor strategies that explicitly recognize regimes (using volume/volatility filters or short-term state classification) or that restrict trading to proven “hot zones” (e.g. opening range, lunch break).  This often leads to hybrid or ensemble architectures.  Critically, *each* mode or regime must be validated separately and jointly to ensure no one mode is carrying all the returns.【41†L223-L234】【27†L69-L77】  

In sum, our guiding principle is: **destroy a strategy with tests before trading it live.**  If it still works, it’s more likely genuine.  We lean on cross-validation techniques and Monte Carlo stress tests adapted for time series, as financial data are non-IID and non-stationary【12†L59-L67】【40†L147-L156】.  Every assumption (costs, event definitions, execution) is treated as a potential Achilles’ heel.

## 2. Train / Validation / Test Structure

We segment historical data **chronologically** into in-sample (training), validation, and out-of-sample (OOS) periods.  The exact splits depend on the system architecture:

- **Rule-Based Strategies:** Treat the strategy parameters as fixed after design.  Use an initial training window (e.g. 1–2 years of ES data) to tune the few rule parameters (range lengths, RSI thresholds, etc.), then **hold out** subsequent data as OOS.  For example, optimize on 2010–2018, then test 2019–2020 without any parameter changes.  One can use a final forward test (e.g. 2021+) for “paper trading” evaluation.  This simple train/test split must include *stressful periods* (e.g. 2008 crash, 2020 COVID spike) to ensure robustness【36†L129-L137】【36†L179-L187】.  If the strategy is re-optimized regularly (e.g. quarterly), we effectively perform a rolling window approach (see next section).  
- **Machine-Learning Components:** These require careful time-series cross-validation.  We **never** do plain k-fold shuffling.  Instead, we use *purged* and *combinatorial purged* CV methods that respect chronology and avoid leakage【40†L147-L156】.  For example, one can divide data into, say, 20 sequential folds; for each, train on all but one fold (with a buffer purge around the test segment) and test on that fold.  The “Combinatorial Purged CV” (CPCV) variant creates many such train/test splits and averages results【40†L147-L156】.  In practice this reduces the Probability of Backtest Overfitting and raises the Deflated Sharpe over naive walk-forward【40†L64-L72】.  We compute PBO by counting how often the validation performance beats the in-sample optimum under many CV permutations【40†L64-L72】; a high PBO (~1) means the model is likely overfit.  All hyperparameter tuning must be done strictly on in-sample CV folds, and the final performance is reported on a **held-out test set** not used in CV.  
- **Mixture-of-Experts / Regime Systems:** These systems (e.g. one model per volatility regime, or a gating network) are validated in a *nested* way.  For each expert, perform the train/val/test splits as above but only on the days belonging to that regime.  Then test the **full composite** system (with the regime classifier) on the OOS period.  It’s crucial that the regime-detection logic itself be computed only from past data.  In other words, if a regime label is based on volatility measures, those measures must use only information up to that time (no peeking ahead).  We typically train the experts on segmented windows (e.g. train a “high-vol” model on past high-volatility days and a “low-vol” model on past low-vol days) and then simulate the system using real-time regime signals on the test set.  This avoids any leakage between regimes or epochs.  

In all cases, we enforce that **no information from the test intervals is used for training or parameter selection**.  Out-of-sample validation must include periods with different characteristics than the training set.  As a rule of thumb, reserve at least 20–30% of data (and ideally 100+ trades) for testing to gain statistical significance【36†L129-L137】.  We also randomize the *order* of validation (e.g. use walk-forward blocks, see below) so that results are not solely from one contiguous block.

## 3. Walk-Forward Design for Intraday ES

For intraday horizons, we use a **rolling (walk-forward) framework** with fixed calendar windows:

1. **Define Window Sizes:**  For example, use the first 2 years as In-Sample (IS) and the next 6 months as OOS, then roll forward by 6 months and repeat.  Choice depends on trade frequency: we want enough data in each IS window to get statistical power (e.g. hundreds of trades).  
2. **Sequential Testing:**  On each iteration, *optimize* strategy parameters on the IS window **only**, then apply those parameters to the adjacent OOS window without changes.  Record the OOS performance and then roll the window forward (dropping the oldest portion).  This mimics a real trader who re-optimizes periodically.  
3. **Purging Overlap:**  To avoid lookahead around boundaries, insert a small purge gap (e.g. 1–2 trading days) between IS and OOS.  This ensures that signals from the end of IS don’t bleed into the start of OOS.  With intraday data, it’s especially important not to overlap by even a few minutes (hence the nested window 12:45–1:00 inside 12:00–1:00 is handled carefully to avoid overlap).
4. **Ensemble CV:**  Instead of a single walk-forward, one can also apply the CPCV method【40†L64-L72】.  For example, divide the whole dataset into many small contiguous folds (say, weeks or months), then perform many random combinations of IS/OOS splits.  CPCV has been shown to yield *lower* overfitting risk (lower PBO) than a simple walk-forward【40†L64-L72】.  

Practically, the walk-forward scheme for ES must respect the trading calendar (avoid roll dates or known low-liquidity days).  For intraday, one might align windows by calendar months or quarters, so that each OOS includes a mix of event types.  The key is that each OOS test should be treated as a **forward simulation** – parameters are frozen once the OOS begins, and performance is logged as if trading live.  Finally, aggregate the walk-forward results: we look for consistent profitability across all OOS slices rather than one big equity curve, as high variance between slices may indicate overfitting【36†L119-L127】.

## 4. Overfitting Metrics and Sensitivity Analysis

To quantify robustness, we use statistical metrics and parameter sensitivity checks:

- **Deflated Sharpe Ratio (DSR):**  Traditional Sharpe can be inflated by multiple trials.  The DSR adjusts Sharpe for *selection bias* and non-normal return distributions【39†L66-L73】.  In essence, after accounting for how many parameter combinations were tested and the skew/kurtosis of returns, the DSR gives the probability that the Sharpe is genuine rather than random.  A DSR below 0 suggests no statistical edge after multiple-testing correction.  In practice, we compute DSR for the best-found strategy as a sanity check on significance【39†L66-L73】.

- **Probability of Backtest Overfitting (PBO):**  We estimate PBO by Monte Carlo cross-validation: shuffle or recombine subperiods many times and see the fraction of strategies whose in-sample Sharpe exceeds their out-of-sample Sharpe.  A PBO close to 0 means the strategy consistently holds up; a PBO near 1 means it usually fails OOS, indicating curve-fitting.  Methods like Purged K-Fold or CPCV inherently produce PBO estimates【40†L64-L72】.  Our rule of thumb is to require PBO ≪ 0.5 before trusting a model.  

- **Parameter Stability (Sensitivity Analysis):**  We systematically vary each parameter around its chosen value.  For a single-parameter rule, we might test ±25% changes; for multi-parameter rules, we build a *sensitivity map*.  For each combination of (parameter ± small increments), we re-run the backtest and record key metrics (Sharpe, drawdown, profit factor).  The goal is to find a **performance plateau**: a broad region of parameter space where metrics remain high【41†L200-L208】.  If the strategy only performs well at a single point (a narrow peak), it is not robust.  Conversely, if Sharpe stays roughly the same for, say, lookback periods 18–22 days, we have a plateau【24†L107-L114】【41†L213-L215】.  We explicitly **visualize** this by heatmaps or 3D plots (performance surface over two parameters) to ensure we have wide sweet spots rather than razor-thin peaks【41†L213-L215】【24†L107-L114】.

- **Monte Carlo / Randomization Tests:**  We also shuffle trade records or randomize minute-by-minute returns (preserving some structure) to generate a distribution of Sharpe under “no-skill” scenarios.  If the real strategy’s Sharpe is well outside this distribution, it is more likely genuine【41†L239-L247】.  For example, randomly varying entry prices by up to 1–2 ticks (or ±a % of ATR) will break any strategy that was accidentally aligned with a particular price point【29†L88-L96】.  If performance collapses under tiny random shifts (or if it survives equally well), that’s a strong indicator of (lack of) robustness【29†L88-L96】【41†L239-L247】.  

By combining these tools (DSR, PBO, sensitivity maps, Monte Carlo), we get a multi-faceted view of overfitting risk.  No single metric is conclusive, but if several red flags emerge simultaneously (e.g. DSR ≲ 0, PBO ≈ 1, narrow parameter peak), we conclude the edge is likely illusory.

## 5. Robustness Across Conditions

A strategy must work (or gracefully fail) under varied market conditions.  We test across: 

- **Volatility Regimes:**  Label days by realized volatility (e.g. VIX ranges or intraday ATR terciles).  Backtest separately on “low-vol” vs “high-vol” subsets.  A robust intraday strategy should not catastrophically fail in one regime unless explicitly designed for it.  For instance, a breakout system might only trade above a volatility threshold; in that case, its performance on low-vol days should be near zero (no signals) rather than a big loss.  If it loses money in the regime it claims to operate in, that’s a bug.  Conversely, if a strategy only makes money in one regime (and flat/zero in others), document that and handle it (e.g. turn the strategy off in unsuitable times).  

- **Macro-Event Days:**  Identify major scheduled events (FOMC, NFP, etc.) or known crises (e.g. “Flash Crash” on May 6, 2010, or COVID sell-off).  Test the strategy’s performance on these days separately.  If **all** the net profit comes from one particular event day or year, that’s suspect.  For example, [41] shows that if a strategy’s worst drawdown is entirely due to a single crisis period, it likely lacks proper risk control【41†L229-L234】.  Similarly, if it only wins big on a Fed day but loses otherwise, the “edge” may not generalize.  We also run an explicit *crash stress test*: simulate conditions such as VIX>40 or liquidity drying up【27†L69-L77】.  For example, we widen spreads by 3× and see if small losses turn into blowups【27†L69-L77】【27†L105-L108】.

- **Hot-Zone Dependence:**  Since our strategy is *designed* to trade only in certain intraday windows (hot zones), we must verify this design choice.  We check performance *outside* those windows as a control.  If the system *accidentally* had an edge only outside its intended hot zones, that indicates overfitting of time filters.  Conversely, if we disable one hot zone at a time in backtests, the P&L drop should be proportional to historical trade frequency.  If disabling, say, the 9:00–11:00 zone eliminates *all* profit, then that zone is doing all the work – we must then test that zone extra hard (it becomes effectively the single-strategy source of returns).

- **Execution & Slippage Assumptions:**  We embed execution models directly in backtests.  For intraday ES, we assume trades occur as market orders or aggressive limit orders with realistic fills.  We then stress-test by **doubling or tripling** assumed slippage.  For example, if typical slippage is 1 tick per fill, re-run with 2–3 ticks slippage.  Similarly, inflate fixed fees (e.g. commission + data + exchange fees) by 50%.  The strategy should remain profitable under these stressed costs; if profits evaporate with modest slippage inflation, the live viability is doubtful.  Prior work advises always including “spread multipliers and slippage buffers” in stress tests【27†L105-L108】.  In practice we might also use synthetic order-book simulations to model adverse liquidity (e.g. during flash crashes) and verify fills.  If the strategy relies on perfect fills (zero slippage) even in low-volume periods, we reject it as unrealistic.【27†L105-L108】【29†L62-L69】

In all these robustness tests, the goal is to surface hidden fragility quickly.  It’s better to kill a strategy in simulation than in live trading.  By explicitly checking different regimes, events, and cost scenarios, we ensure the strategy doesn’t “bake in” a lucky historical quirk.

## 6. Detecting One-Off Dependencies

We look for signs that the strategy’s apparent success hinges on a single niche factor:

- **One Special Year:**  Compute annual metrics (Sharpe, return) year-by-year.  If one year produces a disproportionate share of profits, investigate why.  For example, remove that year from the backtest and see if the strategy still has a positive expectancy.  A drop from Sharpe 1.2 to Sharpe 0.2 by omitting a good year indicates that the strategy may only work under very specific market conditions from that year【36†L179-L187】.  We often do a “year-by-year” cross-validation: train on all years except 1, test on the held-out year (leave-one-year-out).  

- **One Hot Zone:**  Similarly, test the effect of each hot zone.  Disable each time window in turn and measure the impact on performance.  If, say, removing the morning breakout (6:30–8:30) zone cuts P&L by 90%, that zone is doing most of the work.  This isn’t necessarily bad (we *designed* hot zones), but it means the strategy might as well just focus there.  It also means vulnerability: if future market structure changes (e.g. ES liquidity shifts) and that zone stops behaving the same, the strategy could die.  

- **One Event Class:**  Classify trades by event triggers (e.g. high-volume-news days vs. normal days).  Check if trades taken on days with, say, Fed announcements are carrying the edge.  If so, we might need a separate sub-strategy for news days.  Conversely, if the strategy only won big on, say, post-election volatility days, it’s not a general intraday edge.  We often run a backtest *excluding* all known-event days to see if P&L persists.  A strategy should ideally make some money in calm times and in stormy times; complete dependence on rare events suggests a fragile edge.

- **One Fragile Parameter Set:**  Use the sensitivity maps from section 4 to identify if the strategy only works at a very specific parameter combination.  If the “good” region in parameter space is extremely small, the strategy is brittle.  We quantify this by finding how fast performance drops as we move away from the optimal parameters.  If a 10% parameter change causes Sharpe to plummet from 1.2 to 0.2, the design is not reliable【29†L88-L96】【41†L213-L215】.  Ideally we want our chosen parameters to lie in a wide flat region.  

In short, any sign that the strategy’s success is **fragilely tied to a single year, zone, event, or exact setting** is a yellow or red flag.  A viable strategy should degrade gracefully if that edge is removed; if it dies completely, we must reconsider or simplify.  We illustrate this by using cross-sectional tests (leave-out tests) and perturbation analysis: a robust strategy will have a consistent (though perhaps reduced) edge under all such permutations【29†L35-L43】【36†L179-L187】.

## 7. Hierarchy of Tests

Our validation process proceeds in escalating levels of scrutiny:

1. **Sanity Checks:** Verify the fundamentals.  Ensure data integrity (no missing bars, no duplicate timestamps), correct handling of contract rollovers and session breaks, and realistic tick movements.  Confirm the strategy logic makes sense on example days: visually inspect signals on charts to ensure we’re not “buying at future high” due to coding bugs.  Check that all assumptions (e.g. maximum position size, leverage) conform to trading rules.  This step catches trivial errors and blatant lookahead in code.

2. **Baseline Comparisons:** Compare the strategy to simple benchmarks.  For intraday ES, a baseline might be “always buy-and-hold” (cost of carry) or a simple moving-average rule.  If our system can’t beat a naive benchmark (after costs) or a simple heuristic, we have no edge.  Baselines also include comparing to other in-sample subsets (e.g. see if a random-parameter version ever does as well, to gauge luck).  The goal is to ensure the strategy actually *learns something* rather than tapping into a broad market trend.

3. **Perturbation Tests:** As detailed above, we perturb every aspect of the strategy one at a time.  These include changing parameters by ± small amounts【29†L35-L43】, randomizing prices, shifting start dates, and even altering signal phasing (e.g. delay all entries by 1 minute).  A robust strategy shows only gradual performance changes; a sensitive one crashes immediately.  This is one of the most powerful steps to expose overfitting.  We consider any test failure (like a complete loss of edge under a minor tweak) as grounds for returning to design.

4. **Stress/Cost Tests:** We subject the strategy to adverse conditions.  This includes **slippage stress tests** (multiply spreads, add slippage)【27†L105-L108】【29†L62-L69】, worst-case latency, and scenarios of partial fills.  We also test “fat finger” events: e.g. assume an order can go 5 ticks adverse with some probability.  If the strategy’s performance (especially drawdowns or win rate) deteriorates catastrophically under any plausible stress, we classify it as too fragile.  The [27] stress-testing guide emphasizes that adding slippage/spread buffers is essential; if a strategy’s sharp P&L melts away under a bit more slippage, it isn’t deployment-ready【27†L105-L108】.

5. **Walk-Forward / Paper-Trade Checks:** Only after passing all above tests do we run a forward-looking test.  This can be an actual walk-forward simulation on the last segment of data, or a paper-trading (live-simulation) over recent months.  We compare this live-ish performance against the backtest.  The goal is to catch any discrepancy due to market microstructure changes or implementation issues.  Even a well-validated strategy is considered *tentative* until it proves itself in time.  We typically deploy with reduced size and monitor until its P&L curve looks like the worst-case of our backtests; if it suddenly underperforms real-time, we halt and review.

These steps form a funnel: at each stage, strategies that fail are either revised or abandoned.  Only strategies that remain profitable (after all realistic costs) and stable across perturbations advance to the next stage.

## 8. Avoiding Lookahead Bias

We are vigilant about any form of hindsight.  Common pitfalls in intraday context include:

- **Regime Labels:**  If we use labeled regimes (e.g. “this was a high-volatility day”), we ensure labels are based on *data up to that point*.  For example, one might classify a morning as “high-vol” using the ATR of the prior 10 days (known at the morning’s start).  We never retroactively label based on the full day’s realized vol.  Any regime-switching logic must be causal: at time *t*, only use information from times <t.

- **Anchored VWAP & Reference Levels:**  Anchored VWAP or pivots are valid only if their anchor point is in the past.  For instance, an intraday VWAP anchored at the market open is fine, but a VWAP anchored at EOD (end-of-day) is a form of lookahead because it uses volume information that accumulates during the day.  We only anchor VWAP to past events (e.g. previous day’s close, start of current session).  Similarly, any reference like “opening range high” uses only the actual high observed during the opening period, not the day’s eventual high.

- **Event Handling:**  When a calendar event (e.g. Fed statement) occurs at 2:00 pm, a strategy should only incorporate that knowledge at or after 2:00 pm.  For instance, if we have a rule “reverse after Fed,” we code it so that the reversal trigger only fires once the Fed’s price impact is visible.  We do not pre-compute price moves using the eventual outcome of the event.  In short, we timestamp events and only allow the strategy to react to them in a forward-looking manner.

- **Execution Assumptions:**  We never assume instant fills at the next tick or bar.  For example, if a model signals at 9:15:00, we simulate the order hitting the book at 9:15:01, not filling at 9:15:00 price.  This avoids the classic bias of “buy at close of signal bar.”  Instead, we use the *next* bar’s open (or a mid-quote) as the fill price, plus realistic slippage.  This treatment is in line with avoiding lookahead: you can’t trade at a price that hasn’t occurred yet.  

These precautions eliminate the invisible “cheat codes” that can inflate backtest results.  As Michael Harris warns, lookahead bias is insidious because it makes a backtest appear far better than reality【33†L45-L53】.  We double-check by verifying that the equity curve doesn’t become suspiciously smooth or extreme (see below on kill criteria).  Whenever in doubt, we inspect the code to ensure no future data feeds into a decision.  The general rule is: *if a trader living in real time wouldn’t know it yet, the model shouldn’t either*【33†L45-L53】【36†L125-L131】.

## 9. Deployment-Readiness Checklist

Before going live, we require all of the following, with evidence and documentation:

- **Statistical Validity:**  OOS Sharpe (after costs) comfortably above a threshold (e.g. >0.7) and not driven by a handful of trades or singular events.  DSR > 0 (preferably > 0.5).  PBO very low (say <5%).  Cross-validated Sharpe similar to in-sample.  
- **Robustness Confirmation:**  Passed all perturbation and stress tests with modest performance degradation.  No single-point failures identified in section 5–6.  Clear parameter plateaus exist.  Trade-winning probabilities and expectancy remain positive when inputs vary.  
- **Realistic P&L Curve:**  The combined equity curve (walk-forward aggregated or paper-trade) looks like a plausible algorithmic track: choppy, mean-reverting around an uptrend, with drawdowns and volatility commensurate with historical norms.  (In other words, *not* a straight line of small drawdowns).  If the backtest equity was extremely smooth, we check that forward performance (paper or live) actually shows the expected bumps and dips.  
- **Infrastructure Ready:**  Code is reviewed and tested end-to-end.  Data feeds, risk checks, and execution modules are operational.  We simulate a month of live data processing to ensure no surprises.  
- **Risk Controls Implemented:**  Position sizing, maximum loss per trade, daily loss limits, and kill-switch logic are coded.  These controls were in effect during backtests and have been validated.  
- **Paper-Trading Period:**  We require at least 4–12 weeks of live-sim (paper trade) with logs matching (within reason) backtest projections.  The strategy must prove it can handle real-world delays and data quirks.  

Only when every check is green do we allocate real capital.  Even then, we start small and scale up gradually, constantly monitoring that live P&L behaves like the stress-tested scenarios.

## 10. Research Kill Criteria

Finally, we define clear “abort” conditions to stop development on an appealing backtest if it fails reality checks.  Reasons to kill a strategy include:

- **No Statistically Valid Edge:**  DSR ≤ 0 (i.e. Sharpe isn’t statistically significant after deflation) or PBO ≈ 1 (strategy selection is essentially random)【39†L66-L73】【40†L64-L72】.  
- **Unrealistic Performance:**  Sharpe well above 1.5 or near-perfect win rates on backtest【33†L98-L107】.  Such smooth, high returns usually indicate hidden bias.  If the model produces a “too good to be true” equity curve, we stop and find the bug.  
- **Fragile Parameters:**  Absence of any plateau in sensitivity analysis.  If our heatmaps show that only one precise parameter set yields profit (and any tweak causes losses), we scrap it.  A deployable strategy must work across a range of settings【24†L107-L114】【41†L200-L208】.  
- **One-Off Dependence:**  If performance falls apart when one year, zone, or event is removed (i.e. it was carried by that one factor), we kill it or redesign it.  A single-year wonder or a one-hot-zone win is not robust.  
- **Poor Stress-Test Outcomes:**  If the strategy only survives under zero or minimal costs but fails under any realistic slippage or slighly larger commission, it’s not viable【27†L105-L108】【29†L62-L69】.  Similarly, if it is ruined by a moderate market stress scenario (e.g. VIX >30 causes total loss), we drop it.  
- **Regime or Model Breaks:**  If forward testing (paper-trade) shows consistent underperformance compared to backtest, we pause.  Over several weeks, if the strategy does not adapt to new data as it did in backtest, we stop.  

In practice, we assign a tiered set of kill thresholds (e.g. DSR must be >0.5 and out-of-sample Sharpe >0.7).  Failure to meet these kills the project, not just tweaks it.  The goal is to have a **stopping rule** so we don’t waste effort chasing marginal wins.  As the literature warns, no amount of in-sample polishing can make a busted strategy “work” if it fails these harsh checks【33†L98-L107】【36†L179-L187】.  It is far better to kill a strategy early (even if it looks great on some past data) than to deploy a fragile model that blows up with real money.
